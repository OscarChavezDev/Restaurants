ISO/IEC 25023 · Medición de calidad del producto software

Interoperabilidad y tolerancia a fallos del Sistema de Restaurantes de Tingo María

Se aplican las funciones de medición de la norma ISO/IEC 25023 para cuantificar dos subcaracterísticas de calidad del producto software definidas en ISO/IEC 25010 — interoperabilidad (Tabla 8) y tolerancia a fallos (Tabla 17) — sobre el sistema que expone la API REST consumida por los demás sistemas del ecosistema turístico (Eventos, Hoteles, Turismo y Transporte). Cada medida responde a una pregunta concreta mediante una función de medición, en general de la forma X = A/B, donde A y B son conteos o tiempos observables.

MET

Método

sin herramientas externas · reproducible

El sistema bajo prueba es la aplicación web de gestión de restaurantes: backend Java 21 / Spring Boot 3.2 (arquitectura hexagonal, API REST), frontend Next.js 14 y PostgreSQL 16 + PostGIS, orquestados con Docker Compose. La medición corre solo con lo que trae Windows más Docker, de modo que cualquier integrante del equipo puede repetirla con mediciones-iso25023/scripts/*.ps1 (aceptan -BaseUrl para otros entornos). La base técnica que habilita la medición es doble: (a) para interoperabilidad, los endpoints de integración /v1/integration/**, el contrato OpenAPI 3 publicado en /v3/api-docs, el envelope JSON ApiResponse, la autenticación JWT Bearer y CORS; (b) para tolerancia a fallos, el GlobalExceptionHandler, que traduce toda condición de error conocida en una respuesta HTTP controlada con código de error, y la validación Jakarta en los DTO de entrada.

Procedimiento

Procedimiento · Interoperabilidad

medir-interoperabilidad.ps1 actúa como un cliente externo real: hace login JWT con la cuenta administradora del seed, consume los tres endpoints de integración, descarga el contrato OpenAPI y ejecuta un preflight CORS.

Un ítem cuenta para A solo si cumple el criterio documentado: estado HTTP esperado, JSON válido y envelope presente. B = ítems especificados (3 por medida).

Procedimiento · Tolerancia a fallos

medir-tolerancia-fallos.ps1 inyecta 12 patrones de fallo por HTTP. Un fallo cuenta como evitado si el sistema responde con un 4xx controlado — nunca 5xx ni desconexión — y sigue operativo (healthcheck final por Actuator).

Para RFt-3 se mide Ai − Bi con Stopwatch (.NET): del instante de inyección al instante en que la API devuelve el error con su código. Para RFt-2 se inventariaron los servicios y réplicas reales vía Docker Compose.

DIS

Discusión

utilidad del instrumento · limitaciones

El instrumento no se limitó a un ejercicio formal: detectó una deficiencia concreta del producto

La ejecución de RFt-1-G arrojó X = 0,5 (Tabla 5): seis de los doce patrones de fallo inyectados (JSON malformado, identificador inválido, método no soportado, Content-Type incorrecto, parámetro fuera de rango y cuerpo vacío) devolvieron respuestas 500 no controladas, en lugar del 4xx esperado. Los seis casos evitados (violación de validación, credenciales incorrectas, recurso inexistente, ruta inexistente, acceso sin autenticación y token corrupto) sí respondieron con estados 4xx controlados.

Este hallazgo confirma la utilidad del instrumento: permite identificar con precisión qué patrones de fallo no están siendo controlados actualmente, información que de otro modo no sería visible sin una medición sistemática. El detalle exacto del manejo interno de excepciones que provoca estas seis respuestas 500 queda fuera del alcance de esta medición — que se basa en el comportamiento observable de la API desde un cliente externo, no en inspección de código — y se propone como línea de trabajo futuro.

Limitaciones

  • RFt-2-S local da 0 por diseño del despliegue; en producción, Render free sigue siendo un punto único de fallo del backend.
  • Las latencias de RFt-3-S se midieron en localhost; sobre una red real serían mayores.
  • B = 3 ítems por medida de interoperabilidad es un catálogo pequeño; ampliarlo (p. ej. HTTPS/TLS en producción) lo haría más robusto.
  • Una sola corrida por medida; repetir y promediar fortalecería la validez estadística de RFt-3-S.
  • Y más importante: los seis casos con respuesta 500 (Tabla 5) representan una deficiencia real del sistema aún no corregida al momento de esta medición, y esta medición no permite determinar su causa exacta a nivel de código.

Trabajo futuro

  • Investigar a nivel de código la causa de las seis respuestas 500 no controladas y corregir el manejo de excepciones correspondiente; luego volver a ejecutar la batería para verificar si RFt-1-G alcanza X = 1,0.
  • Replicar las pruebas de interoperabilidad como colección de Postman ejecutada con Newman, herramienta ampliamente reconocida, manteniendo los scripts de PowerShell como verificación cruzada.
  • Ejecutar tres corridas de RFt-3-S y reportar media y desviación estándar.
  • Medir la subcaracterística de disponibilidad (Tabla 16) con la infraestructura ya preparada en el repositorio.
CON

Conclusiones

El sistema demuestra interoperabilidad plena (X = 1,0 en las tres medidas de la Tabla 8). En tolerancia a fallos, la medición reveló una deficiencia real: solo seis de los doce patrones de fallo fueron evitados (RFt-1-G = 0,5), con seis respuestas 500 no controladas correspondientes a JSON malformado, identificador con tipo inválido, método HTTP no soportado, Content-Type incorrecto, parámetro fuera de rango y cuerpo vacío. Aun así, el sistema notifica sus fallos con rapidez (0,079 s en promedio, RFt-3-S) y el servicio permaneció operativo tras toda la batería. La principal oportunidad de mejora es doble: por un lado, investigar y corregir el manejo de excepciones responsable de los seis patrones no evitados; por otro, atender la redundancia de componentes (RFt-2-S), nula en el entorno local y parcial (0,67) en producción. Los scripts de PowerShell, los archivos CSV de resultados y los criterios de evaluación documentados hacen posible repetir la medición sin intervención manual y verificar en una futura entrega si las correcciones propuestas elevan RFt-1-G a un nivel aceptable; los resultados completos, con fórmulas que se recalculan al editar los datos, están disponibles en el libro Excel que acompaña a este informe.
